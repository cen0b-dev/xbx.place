# xbx.place

Vite + TypeScript static frontend for browsing Xbox 360 titles and opening external mirror download links.

## Local dev

```bash
npm ci
npm run dev
```

For **download buttons** (proxy mode), also run **`npm run proxy`** in another terminal (or set `VITE_DOWNLOAD_PROXY_ORIGIN` to a deployed worker). With only `npm run dev`, Vite forwards `/download` to the local proxy on port 8787.

## Refresh mirror data

```bash
npm run sync:romsworlds
```

This command scrapes the romsworlds Xbox 360 catalog and writes:

- `data/romsworlds-index.json` (canonical local data source)
- `public/master_index.json` (frontend runtime dataset)

## Build x360db catalog for site UI

```bash
npm run build:x360db-catalog
```

This builds `public/master_index.json` from x360db title IDs so artwork URLs resolve correctly, and attaches IA downloads when filenames match entries in `public/ia-file-map.json`.

By default it outputs **Redump-only visible titles** (titles with at least one matched IA download). To include all x360db titles:

```bash
X360DB_REDUMP_ONLY=0 npm run build:x360db-catalog
```

## Build Internet Archive redump map (cookie auth)

```bash
npm run build:ia-map
```

The IA map tool:

- reads IA identifiers from [r-roms Xbox 360 source](https://r-roms.github.io/Microsoft/microsoft-xbox360)
- limits to `Redump.org` buckets only
- reads credentials from **`IA_COOKIE_POOL`** (JSON array `[{"user":"...","sig":"..."}, ...]` — set in `.env.local` or the shell; do not commit)
- optional obfuscated env: **`IA_COOKIE_POOL_B64`** (base64 of the same JSON; supports multiple rounds via `IA_COOKIE_B64_ROUNDS`)
- fallback single account via `IA_LOGGED_IN_USER` and `IA_LOGGED_IN_SIG`
- writes `public/ia-file-map.json`

Optional env flags:

- `IA_MAP_LIMIT_IDENTIFIERS=3` (test with first N redump buckets)
- `IA_MAP_FILTER_TO_MASTER=1` (map only filenames already present in `public/master_index.json`)

## Build

```bash
npm run build
```

`npm run build` automatically runs `npm run build:data` first so deploys include fresh catalog data.

## GitHub Pages deploy

This repo includes `.github/workflows/deploy-github-pages.yml`.

1. Push to `main`.
2. In GitHub repo settings, set **Pages** source to **GitHub Actions**.
3. The workflow builds with Vite and deploys `dist`.

## Data files

- `public/master_index.json` is loaded by the frontend.
- `data/romsworlds-index.json` is the canonical mirror index generated by the sync script.
- Cover art uses the x360db raw GitHub source URL at runtime.
- Mirror links target `file.romsworlds.com`.

## One-download lock

- The frontend enforces one active download at a time per browser profile.
- Users can clear that lock with the "Cancel active download lock" button in the header.

## Downloads: local vs GitHub Pages (no paid proxy)

GitHub Pages only serves **static files** — it cannot run `server/proxy.example.ts`. Hiding downstream URLs needs **some** server that forwards bytes. Here is the **no-cost** mapping:

### Local development

1. Terminal A: **`npm run proxy`** — reads `public/master_index.json`, listens on port **8787**.
2. Terminal B: **`npm run dev`** — Vite proxies **`/download` → http://127.0.0.1:8787** (see `vite.config.ts`).
3. Leave **`VITE_DOWNLOAD_PROXY_ORIGIN` unset**: the app uses relative **`/download?key=…`**, which hits the Vite dev server and is forwarded to your local proxy.

### Production (GitHub Pages)

Use the **Cloudflare Workers free tier** worker in `workers/download-proxy/`:

1. Edit `workers/download-proxy/wrangler.toml` and set **`MASTER_INDEX_URL`** to your live catalog, e.g. `https://<you>.github.io/xbx.place/master_index.json`.
2. Deploy: `cd workers/download-proxy && npx wrangler deploy`
3. Store IA cookies with **`npx wrangler secret put IA_COOKIE_POOL`** (one-line JSON array, same shape as local env), or **`IA_COOKIE_POOL_B64`** + optional **`IA_COOKIE_B64_ROUNDS`**. You can also set **`IA_LOGGED_IN_USER` / `IA_LOGGED_IN_SIG`** in the dashboard (avoid committing real values in `wrangler.toml`).
4. In the GitHub repo: **Settings → Secrets and variables → Actions → Variables** — add **`VITE_DOWNLOAD_PROXY_ORIGIN`** = your worker URL (e.g. `https://xbx-place-download-proxy.<account>.workers.dev`, no trailing slash).
5. Re-run the Pages workflow (or push) so the built `dist` embeds that origin.

The worker **streams** the upstream response (no full-file buffer in the worker) and applies the same host allowlist + optional IA cookies as `server/proxy.example.ts`.

### Optional: self-hosted Node proxy

For a VPS or homelab, run `npm run proxy` (or `tsx server/proxy.example.ts`) and point **`VITE_DOWNLOAD_PROXY_ORIGIN`** at that origin.

### Reference: Node proxy behavior (`server/proxy.example.ts`)

- `GET /download?key=<downloadFilename>` (browser never sees the raw mirror / IA URL in the page)
- Host allowlist via **`ALLOWED_DOWNLOAD_HOSTS`**
- IA: random account from **`IA_COOKIE_POOL`** / **`IA_COOKIE_POOL_B64`** or **`IA_LOGGED_IN_USER` / `IA_LOGGED_IN_SIG`**
- cookie values are sanitized and canonicalized before building `Cookie` headers to reduce header-injection risk

## Cookie expiry reality

- Archive session cookies (`logged-in-sig`) expire and cannot be made permanent client-side.
- Best practice: keep multiple accounts as entries in **`IA_COOKIE_POOL`**, monitor failures in your proxy/logs, and refresh expired values in secrets / `.env.local` only (never commit).
